## 总体架构

Edge Image Gateway 是一个运行在 Cloudflare Workers 上的无服务器边缘应用程序，使用 GitHub 仓库作为后端存储层，Cloudflare D1 作为主数据库（元数据、配置和路径索引），KV 仅用于速率限制、GitHub API 监控和通知节流。

## 请求处理流程

> 详细的请求生命周期（中间件链、图片处理回环、上传全链路、分享链接验证）见 [架构总览 - 请求生命周期](architecture-overview.md#3-请求生命周期)。

### 中间件链（概要）

中间件按顺序执行，任何中间件返回响应则终止链：

| 中间件 | 文件 | 职责 |
|--------|------|------|
| 速率限制 | [rateLimit.ts](../src/middleware/rateLimit.ts) | IP 级别令牌桶限流 + 404 封禁 |
| 防盗链 | [referer.ts](../src/middleware/referer.ts) | Referer 白名单校验 + Sec-Fetch-Dest 智能放行 |
| 签名认证+熔断 | [signature.ts](../src/middleware/signature.ts) | HMAC-SHA256 签名验证 + 紧急熔断 + 分级路径防御 |
| 管理认证 | [adminAuth.ts](../src/middleware/adminAuth.ts) | API Token / Cloudflare Access / TOTP 认证（仅 `/admin` 路径） |

### 路由分发逻辑

路由引擎 [repoRouter.ts](../src/services/repoRouter.ts) 是多仓库的核心路由引擎：

- **D1 缓存** — 30 秒内内存缓存仓库元数据，减少 D1 读取
- **路径索引** — 先查精确路径索引，再查前缀匹配，再查路由规则
- **读路由** — 按 `ReadRule[]` 前缀匹配，未匹配则使用当前写仓库
- **写路由** — 自动选择有剩余空间的活跃仓库，超容量自动切换
- **兜底策略** — KV 不可用时回退到环境变量中的默认仓库

### 2. GitHub 存储服务

[github.ts](../src/services/github.ts) 封装了所有 GitHub API 调用：

| 方法 | 用途 |
|------|------|
| `fetchRaw` | 获取文件原始内容 |
| `getFile` | 获取文件元数据 |
| `fileExists` | 检查文件是否存在 |
| `putFile` | 上传/更新文件 |
| `deleteFile` | 删除文件 |
| `getTree` | 获取仓库目录树 |
| `createRepository` | 创建新仓库 |

### 3. 管理面板 API

管理面板由多个子路由组成：

| API 模块 | 文件 | 功能 |
|----------|------|------|
| Repos | [repos.ts](../src/routes/admin/api/repos.ts) | 仓库 CRUD、状态管理 |
| Files | [files.ts](../src/routes/admin/api/files.ts) | 文件列表、目录树、删除 |
| Upload | [upload.ts](../src/routes/admin/api/upload.ts) | 单文件/批量上传 |
| Stats | [stats.ts](../src/routes/admin/api/stats.ts) | 统计和监控数据 |
| Audit | [audit.ts](../src/routes/admin/api/audit.ts) | 审计日志查询 |

### 4. 缓存策略

多级缓存提升性能：

| 缓存层级 | 说明 | TTL |
|----------|------|-----|
| Workers Cache (Cloudflare CDN) | 成功响应的边缘缓存 | 默认 7 天（可配置） |
| 404 缓存 | 防止频繁请求不存在的文件 | 60 秒 |
| 缓存变体 | 带查询参数（w, h, q, fit）的请求独立缓存 | 与主缓存相同 |
| 内存缓存 | 仓库元数据内存缓存 | 30 秒 |

**缓存控制头：**

```
Cache-Control: public, max-age=604800, s-maxage=604800, immutable
```

## D1 键设计

| 表名 | 用途 | 说明 |
|------|------|------|
| `repos` | 仓库元数据 | 所有仓库的配置、容量、状态 |
| `paths` | 路径索引 | 文件路径 → 仓库映射，去重哈希 |
| `auth_tokens` | 认证令牌 | API Token 管理（权限、路径前缀、有效期） |
| `audit_logs` | 审计日志 | 操作记录（上传、删除、配置变更） |
| `system_config` | 系统配置 | 路由规则、写仓库 ID、功能开关等 |
| `migration_tasks` | 迁移任务 | 文件跨仓库移动/重命名任务状态 |

## KV 键设计（仅限以下用途）

KV 目前已**不再承担数据存储职责**，仅用于以下三种特定场景：

| 键模式 | 值类型 | 用途 | TTL |
|--------|--------|------|-----|
| `ban::{ip}` | `string` | IP 封禁（404 超限后封禁 5 分钟） | 300s |
| `err404::{ip}::{minuteBucket}` | `string` | 404 错误计数（仅内存，超阈值才写 KV） | 120s |
| `github_rate::{repoId}` | `JSON` | GitHub API 剩余配额监控 | 无 |
| `alert_sent::{key}` | `string` | 告警通知节流（同一条告警 N 小时内不重复发送） | 按 hours 配置 |

> 注意：`backfill.ts` 中的 `repo::*`、`path::*`、`auth::token::*`、`route::current_write` 等 KV 读操作仅存在于**旧数据迁移回填**端点，不属于运行时数据流。正常运行时所有数据读写均通过 D1。

              }

## Cron 定时任务

系统注册了定时触发器，用于自动同步各仓库的容量统计：

- **触发频率**：可配置（建议每 6 小时）
- **任务内容**：遍历所有仓库，从 GitHub API 获取实际文件数和大小
- **更新 D1**：将统计结果写入 `repos` 表的 `used_bytes` 和 `file_count` 字段（通过 `dbService` 直接更新 D1）
- **触发告警**：容量接近上限时发送 Telegram 通知

## D1 主存储与 KV 缓存（降级）

系统采用 **D1-first** 架构：D1 是所有数据的唯一主存储（source of truth），KV 仅用于三种特殊场景（速率限制、GitHub API 监控、告警节流），不再作为数据镜像。

### 架构原则

1. **D1 是主存储** — 所有仓库元数据、路径索引、认证令牌、审计日志、系统配置、迁移任务均存储在 D1 表中
2. **KV 不是数据镜像** — 不再维护与 D1 对应的 KV 副本，且减少写成本
3. **KV 的用途限定** — 仅限 `rateLimit.ts`（IP 封禁/404 计数）、`github.ts`（API 速率监控）、`notifications.ts`（告警节流）
4. **KV 降级路径** — 在 D1 不可用时的兜底查询仍然保留（如 `repoRouter.ts` 中 D1 失败后回退到 KV），但这属于降级策略，非正常流程

### 写入策略

```
写入请求
    │
    └──► 写入 D1（主存储）
            │
            ├── 成功 ──► 返回成功
            │
            └── 失败 ──► 记录错误日志，返回错误
            │
            （KV 不再写入数据镜像）
```

### 读取策略

```
读取请求
    │
    ├──► 1. 查询 D1（主存储）
    │       │
    │       ├── 命中 ──► 返回数据
    │       │
    │       └── 未命中/失败 ──► 2. 查询 KV 降级（仅限路径映射）
    │                              │
    │                              ├── 命中 ──► 返回数据
    │                              └── 未命中 ──► 路由规则 / 兜底仓库
    │
    └──► 核心功能不依赖 KV，KV 不可用时仍可工作
```

### 一致性保证

| 场景 | 说明 |
|------|------|
| 正常写入 | D1 原子事务保证路径和容量统计的一致性 |
| KV 故障 | 不影响核心功能，仅速率限制和告警节流降级 |
| D1 故障 | 降级到 KV 路径映射（仅剩余旧数据），新写入不可用 |
| 并发写入 | D1 使用相对更新（`used_bytes = used_bytes + ?`）避免竞态 |

### 数据修复

迁移完成后，KV 中的数据仅为历史存量。如需将 KV 旧数据（`repo::*`、`path::*`、`auth::token::*`）回填到 D1，可通过管理面板的回填端点：

```bash
# 手动触发回填（仅用于首次迁移或数据修复）
curl -X POST https://{你的域名}/admin/api/backfill/start \
  -H "Authorization: Bearer <token>"
```

> 正常运行时无需执行此操作，所有新数据已直接写入 D1。

## 核心数据流

### 图片上传全链路

```
客户端                                      Worker                                   GitHub
  │                                           │                                        │
  │  POST /upload (multipart/form-data)        │                                        │
  │──────────────────────────────────────────►│                                        │
  │                                           │                                        │
  │                                           │  1. 中间件：速率检查、防盗链、签名       │
  │                                           │  2. 解析 multipart，提取文件和路径       │
  │                                           │  3. 计算 SHA-256 哈希                   │
  │                                           │  4. 剥离 EXIF 元数据（JPEG/PNG）        │
  │                                           │  5. 检查文件类型和大小限制               │
  │                                           │                                        │
  │                                           │  PUT /repos/{owner}/{repo}/contents/{path}
  │                                           │───────────────────────────────────────►│
  │                                           │                                        │
  │                                           │  ◄─── 201 Created (sha, size) ────────│
  │                                           │                                        │
  │                                           │  6. 写入 D1 paths 表（路径索引）         │
  │                                           │  7. 更新仓库容量统计（相对更新）          │
  │                                           │  8. 写入审计日志                         │
  │                                           │                                        │
  │  ◄── 200 OK { url, sha, size, repo } ────│                                        │
  │                                           │                                        │
```

### 图片读取全链路（含缓存）

```
客户端                                      Worker                                   GitHub
  │                                           │                                        │
  │  GET /images/photo.jpg?w=200              │                                        │
  │──────────────────────────────────────────►│                                        │
  │                                           │                                        │
  │                                           │  1. 中间件：速率检查、防盗链             │
  │                                           │                                        │
  │                                           │  2. 检查 Workers Cache (L1)             │
  │                                           │     ├── 命中 → 直接返回                  │
  │                                           │     └── 未命中 ↓                        │
  │                                           │                                        │
  │                                           │  3. 解析处理参数 (w=200)                │
  │                                           │                                        │
  │                                           │  4. 检查 R2 Cache (L2)                  │
  │                                           │     ├── 命中 → 回填 L1 → 返回            │
  │                                           │     └── 未命中 ↓                        │
  │                                           │                                        │
  │                                           │  5. 路由解析：确定目标 GitHub 仓库        │
  │                                           │                                        │
  │                                           │  GET /repos/{owner}/{repo}/contents/{path}
  │                                           │───────────────────────────────────────►│
  │                                           │                                        │
  │                                           │  ◄─── 200 OK (raw content) ───────────│
  │                                           │                                        │
  │                                           │  6. 触发 Image Resizing (cf.image)      │
  │                                           │  7. 写入 R2 缓存 (L2)                   │
  │                                           │  8. 写入 Workers Cache (L1)             │
  │                                           │                                        │
  │  ◄── 200 OK (processed image) ───────────│                                        │
  │                                           │                                        │
```
