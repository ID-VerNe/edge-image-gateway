## 核心模块说明

### 1. 应用入口（[index.ts](../src/index.ts)）

- 使用 Hono 框架创建应用实例
- 注册全局中间件（限流 → 签名 → 防盗链）
- 挂载管理后台路由（`/admin/*`）
- 注册图片请求处理路由（`/*`）
- 注册定时任务（`scheduled` 事件，每分钟同步仓库容量）

### 2. 中间件链路

请求按以下顺序经过中间件：

1. **限流中间件**：基于 `CF-Connecting-IP` 计算请求频率，超出限制返回 429。同时追踪 404 请求次数，超过阈值自动封禁。
2. **签名验证中间件**：对需要签名的路径（`/private/`、`/draft/`、`/raw/`）或紧急熔断状态下，验证 `sig` 和 `exp` 参数。
3. **防盗链中间件**：检查 `Referer` / `Origin` 请求头，验证是否在白名单中。

### 3. 图片请求处理（[image.ts](../src/routes/image.ts)）

核心流程：

```
请求进入
  ↓
路径安全检查（禁止 `..` 路径穿越）
  ↓
多仓库路由解析（根据路径前缀匹配仓库）
  ↓
缓存查询（Cloudflare Cache API）
  ├─ 命中 → 直接返回缓存响应
  └─ 未命中
       ↓
     从 GitHub Contents API 流式获取文件
       ↓
     判断是否需要缩放（是否有 w/h/q/fit 参数）
       ├─ 否 → 直接返回原始文件
       └─ 是
            ↓
          回环代理：构造带内部签名的请求回调自身
            ↓
          Cloudflare Image Resizing 处理缩放
            ↓
          缩放成功 → 返回缩放后图片
          缩放失败 → 回退返回原始图片
       ↓
     写入 Cloudflare Cache
      ↓
     返回响应（剥离后端标识头）
```

### 4. 多仓库路由（[repoRouter.ts](../src/services/repoRouter.ts)）

- 通过 Cloudflare KV 存储多个仓库的元数据
- 支持基于路径前缀的读路由规则
- 支持写仓库切换
- 30 秒内存缓存 + KV 回退
- 无 KV 时优雅降级到单仓库模式

### 5. 回环代理原理

这是本项目的核心技术亮点：

```
用户请求: /image.png?w=800
  ↓
Worker 检测到需要缩放
  ↓
生成内部签名（使用 `SIGN_SECRET`）
  ↓
构造回调请求: http://internal/{path}?_int_sig=xxx&_int_exp=yyy
  ↓
Worker 重新处理回调请求（跳过中间件，直接返回原始文件）
  ↓
Cloudflare Image Resizing 引擎捕获回调响应 → 执行缩放 → 返回结果
```

**为什么需要回环代理？**

Cloudflare Image Resizing 需要直接访问原始图片 URL。但图片存储在 GitHub 私有仓库中，需要 `GITHUB_TOKEN` 认证才能访问。Image Resizing 引擎无法持有 GitHub Token，因此无法直接从 GitHub 拉取图片。

通过回环代理，Worker 自身作为"认证代理"去 GitHub 获取原始图片，然后 Image Resizing 引擎在 Worker 响应流上执行缩放操作，完美绕过了这一限制。

## 数据流

### 图片读取流程

```
用户 → Cloudflare Worker → GitHub API → 原始图片 → 回环缩放 → 缓存 → 用户
```

### 图片上传流程（通过管理后台）

```
用户上传 → Admin API → GitHub Contents API (PUT) → 写入仓库 → 返回 URL
```

### 定时同步流程

```
Cron 触发 → 遍历所有注册仓库 → 调用 GitHub API 统计容量 → 写入 KV
```

## 瘦缓存层

项目采用"瘦 Source + 厚 Cache"的架构理念：

- **GitHub 仓库**仅作为持久化存储层，不承担流量分发
- **Cloudflare Cache** 是关键的性能支柱，绝大多数请求直接从边缘缓存命中
- 缓存默认 TTL 为 7 天（可通过 `CACHE_TTL_SECONDS` 配置），且遵循缓存分层原则

```
用户请求
  ↓
Cloudflare 边缘缓存（CDN 层）
  ├─ HIT → 直接从边缘节点返回，零回源延迟
  └─ MISS
       ↓
    Worker 处理（计算层）
       ↓
    GitHub API 回源（存储层）
```

缓存键基于完整 URL（包括查询参数），因此 `?w=200` 和 `?w=800` 被视为不同的缓存条目，各自独立缓存。

## 安全设计

本项目采用七层纵深防御体系：

1. **限流**：最外层防护，阻止高频请求和 DDoS
2. **签名验证**：验证请求合法性，防止未授权访问
3. **防盗链**：验证来源域名，防止外部盗链
4. **目录分级**：对敏感路径强制签名保护
5. **紧急熔断**：极端情况下的全站保护
6. **响应脱敏**：隐藏后端架构信息
7. **路径安全**：防止路径穿越攻击

详细配置请参阅 [安全指南](./security.md) 和 [配置参考](./configuration.md)。
