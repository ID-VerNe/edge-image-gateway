### **方案已达到可落地标准,剩下的是几个执行期的边界细节**

整体可行,可以开工。下面这些不是"拦路"问题,而是上线前最好想清楚、否则中途容易返工的点。

### **双写阶段:这版最大的进步,但要补两个一致性细节**

你用"Phase 1 双写 + Phase 1.5 读切换 + 观察期"替代了原来的"脚本一次性迁移",这彻底解决了上一轮我担心的"迁移窗口丢写"问题,思路很对。但双写本身引入了一个新的边界情况要处理:

**第一,双写的失败语义要定义清楚。** 当一次上传同时写 KV 和 D1,如果 KV 成功、D1 失败(或反过来),你怎么处理?有三种选择:

- **以 KV 为准(推荐):** Phase 1 阶段 KV 仍是真相源,D1 写失败只记日志告警、不阻断请求。这样不影响线上,又能在观察期通过对比发现 D1 写漏。
- 以 D1 为准:此时 D1 还没验证过,风险太大,不建议。
- 两边都失败才报错:会让 D1 的不成熟拖累线上可用性。

明确"Phase 1 以 KV 为准、D1 失败不阻断"这一条,你的"瞬时回滚"承诺才真正成立。

**第二,Backfill 与双写的执行顺序有竞态。** 你的步骤是"先开双写(step 3)、再跑 backfill(step 4)"——顺序是对的(先双写能保证 backfill 期间的新数据不漏)。但 backfill 脚本往 D1 插历史数据时,可能和双写正在插的同一条记录撞上。所以 **backfill 的写入必须用幂等的 `INSERT ... ON CONFLICT DO UPDATE`(UPSERT)**,而不是裸 `INSERT`,否则主键冲突会让脚本中断或产生脏数据。`paths.path` 既然是唯一键,正好天然支持 UPSERT。

### **D1 事务:注意 D1 的事务模型限制**

你把 `repos.used_bytes` 更新和 `paths` 增删放进单事务,方向完全正确。但有个 D1 特有的限制要知道:

**D1 目前不支持跨多个 HTTP 往返的交互式事务(interactive transaction),它依赖 `batch()` 把多条语句作为一个原子单元提交。** 也就是说,你不能"先 SELECT 查出当前 used_bytes,在 JS 里加一下,再 UPDATE"——这中间隔了往返,不在同一原子单元里。正确做法是用**相对更新**让数据库自己算:

```sql
UPDATE repos SET used_bytes = used_bytes + ?, file_count = file_count + 1 WHERE id = ?;
```

把这条 `UPDATE` 和 `paths` 的 `INSERT` 放进同一个 `db.batch([...])` 提交,才是真正的原子。如果用"读-改-写"的方式,高并发上传同一仓库时会丢更新(race condition),`used_bytes` 会越统计越不准。这是从 KV 思维迁到 SQL 时最容易踩的坑,务必注意。

### **R2 缓存:还有两个没覆盖到的小风险**

三级缓存梯度、key 规范化、按创建时间(非 LRU)淘汰、in-flight 单飞——这些你都点到了,很完整。再补两个:

**第一,异步写 R2 要用 `ctx.waitUntil()`,别阻塞响应。** 缓存未命中时,正确流程是:回源 GitHub → 处理 → **立即把结果返回给客户端** → 然后 `ctx.waitUntil(R2.put(...))` 在后台异步写缓存。如果你同步 `await R2.put` 再返回,会白白增加首次请求的延迟。这个细节没写进 plan,但实现时很关键。

**第二,单飞(Singleflight)在 Workers 上是"每实例"的,不是全局的。** Cloudflare Workers 是多实例隔离的,你在一个 Worker 实例里用内存 Map 做 in-flight 去重,只能挡住**同一实例**内的并发重复请求。当 R2 缓存集中过期、海量请求被负载均衡打到**几十个不同实例**上时,每个实例都会各自回源一次 GitHub。所以单飞能把"成千上万次回源"压到"几十次",但压不到"1 次"。

这通常够用了,但你的验证项里写了"verify in-flight dedup prevents GitHub rate limits"——要把预期设对:它是**大幅削峰**,不是**完全杜绝**。如果你的图片量大到几十次并发回源仍可能触发 GitHub 限流,真正的"全局单飞"需要借助 Durable Objects(单点协调)或给热点图片错开过期时间(给 Lifecycle TTL 加随机抖动,避免同一时刻雪崩式过期)。**给缓存过期时间加 jitter** 是最低成本的缓解手段,建议加上。

### **R2 也有写限额,Phase 2 的监控别只盯存储**

你在监控项里关注了"D1/R2 usage 在免费额度内",但具体到 R2 要分清两类限制:**存储 10GB** 你抓住了,但 **Class A 操作(含 `PUT`,即写缓存)免费额度是 100 万/月**。每次缓存未命中都会触发一次 R2 `PUT`,如果你的图片访问长尾很长、命中率低,加上 Lifecycle 频繁过期导致反复回源重写,`PUT` 次数可能比想象中涨得快。监控时把 **R2 Class A 操作数**单独列一个观测指标,别只看存储容量。

这版 plan 我认为**可以直接进入实施**,核心架构和迁移安全性都立住了。落地时把这几条执行细节钉死就稳了:
